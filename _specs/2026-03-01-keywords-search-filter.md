# Spec for Keywords Search and Filter

branch: claude/feature/keywords-search-filter
instructions: @.claude/instructions/vuejs3.instructions.md

## Summary

Recipes in Firestore now have a `keywords` field (array of strings) generated by the Ollama enricher. This feature replaces the auto-derived `tags` system with `keywords` across the browse and ingredient search pages â€” for filtering, display on recipe cards, and the tag/keyword filter modal.

The `tags` field was auto-derived on the client from recipe name + description + ingredients via a stopword filter. `keywords` are richer, AI-generated labels (e.g. cuisine type, meal occasion, ingredient type) stored directly on the recipe document. Recipes without a `keywords` field should fall back to `tags` for backwards compatibility during the transition period.

## Functional Requirements

- In `useRecipeSearch.js`, read `r.keywords ?? r.tags ?? []` instead of `r.tags ?? []` when computing `topTags` and applying tag/keyword filters.
- In `useIngredientSearch.js`, apply the same `keywords ?? tags` fallback when filtering by selected keywords.
- Rename all user-facing labels from "Tags" to "Keywords" in the browse view, ingredient search view, and the filter modal.
- Rename the component `TagFilterModal` to `KeywordFilterModal` (or update its internal label text at minimum) to reflect the new terminology.
- In `RecipeCard.vue`, display `keywords` (falling back to `tags`) instead of only `tags`. Label or aria-label should say "keyword" not "tag".
- The search input in `useRecipeSearch.js` should also match against `keywords` in addition to `r.name`, so a keyword like "Italian" will surface matching recipes even if the word is not in the recipe name.
- Accessible form labels and ARIA attributes where appropriate (WCAG 2.1 Level AA standard).
- Responsive layout suitable for mobile and desktop.

## Possible Edge Cases

- Recipes without an `keywords` field (older documents not yet enriched) must gracefully fall back to `tags` so they remain discoverable.
- Recipes with neither `keywords` nor `tags` should return an empty array and not break filtering logic.
- The keyword filter set should be cleared when navigating away and re-entering the page (existing behaviour should be preserved).
- Duplicate keywords across `keywords` and `tags` arrays (in fallback scenario) should be de-duplicated before display.

## Open Questions

- Should the `tags` field be fully removed from the recipe form submission (`useRecipeForm.js`) now, or kept until all recipes are enriched? (Recommendation: keep `tags` generation for now as a fallback; remove in a future cleanup pass.)
- Should the text search (`useRecipeSearch`) match against `keywords` in addition to `name`, or only use keywords for filter chips? (Recommendation: match both â€” include keywords in the searchable text.)

## Testing Guidelines

Create test file(s) in `src/tests` for the new feature. Cover the following without going too heavy:

- `useRecipeSearch` returns `keywords` from a recipe that has them, and falls back to `tags` when `keywords` is absent.
- Filtering by a keyword correctly includes/excludes recipes.
- Text search matches a recipe whose `keywords` contains the search term but whose `name` does not.
- A recipe with neither `keywords` nor `tags` does not break the composable.
